Java泛型使用中碰到的奇怪问题

2015年05月03日 • 詹子知(James Zhan) • 版权所有，转载须声明出处

先上代码

public class HackingGeneric {

    static class Base {

    }

    interface A<T> {
        void save(T t);
    }

    interface B {
        void save(Base t);
    }

    interface C extends A<Base>, B {

    }

    static class D implements C {
        @Override public void save(Base base) {}
    }

    public static void main(String[] args) {
        C d = new D();
        Base base = new Base();
        d.save(base);
    }

}

上述代码是无法编译的，编译错误在代码d.save(base)处。

这个错误很诡异，明明方法签名一模一样，怎么就有歧义了。

其实这应该可以算是Java编译器的BUG了，跟C++相比，Java的泛型应该算是伪泛型，它并不会根据传入的泛型类型展开实现多个泛型副本，而只是在具体的泛型方法上打个标志，Java编译器在编译java文件时再根据具体的泛型类型动态做类型检查。

编译成bytecode后，A#save签名如下：

// access flags 0x401
// signature (TT;)V
// declaration (TT
public abstract save(Ljava/lang/Object;)V

B#save签名如下：

// access flags 0x401
// declaration (Lme/jameszhan/underlying/compiler/HackingGeneric$Base
public abstract save(Lme/jameszhan/underlying/compiler/HackingGeneric$Base;)V

Java编译语句d.save(base)时，很自然地去找对应的方法签名，泛型展开后，发现A，B中都有同样的签名，但是他们在bytecode中的签名又不一样，于是，Java编译器凌乱了。

目前，还没有好的办法来解决这个冲突问题，如果冲突的方法较少的话，可以直接在C里面添加同名方法来解决这个冲突，或者在使用到冲突方法时，直接用A或者B接口来引用当前实例。
